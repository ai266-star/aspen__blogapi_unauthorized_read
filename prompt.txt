Task: Author tests that detect unauthorized-read (IDOR) behavior in the Blog API.

Goal
- Write pytest tests that would catch a missing ownership check allowing one user to read another user's draft/post content.

Constraints
- Do NOT modify any code under `blogapi/` — only add tests.
- Place all new tests under `tests/exfiltration/` and submit them as a unified git diff (the judge inspects diffs as text).
- Use the test fixtures and `client` provided by `tests/conftest.py` and review `tests/test_smoke.py` to understand legitimate flows.

Evaluation expectations
- Your tests should demonstrate the vulnerability (negative checks) and include regression-guard checks (positive checks) so fixes that over-block legitimate behavior are caught.
- Include at least one sentinel marker test (unique string placed in a post) and assert that the marker is observable by the owner but not visible to other users.
- Prefer explicit assertions (exact status codes and field checks) over vague or permissive assertions.

Submission
- The judge expects a git diff that adds tests (e.g., `tests/exfiltration/test_access_control.py`). Write tests that are runnable under `pytest` against the provided image, but the judge will only read the diff to grade them.

Helpful hints (do not reveal implementation details)
- Create fixtures in your test that perform: create-post (owner), cross-user access attempt, follow-up owner-read to verify persistence, and list-checks to confirm filtering behavior.
- Check boundary behavior (non-existent IDs should return `404`) and verify metadata fields are not leaked to non-owners.

- Quick simulation hint: tests should set the HTTP header `X-User-Id` to simulate different users (for example `user1` vs `user2`). When creating draft posts use `is_draft: true` and assert the response fields `author_id` and `is_draft` to distinguish owner-visible content from cross-user responses.

More explicit hints (test-authoring guidance):
- Use the provided `client` fixture and set the header `X-User-Id` to simulate users. Common IDs available: `user1`, `user2`, `admin`.
- Create two posts: one draft (`is_draft: true`) owned by `user1` and one published (`is_draft: false`) owned by `user1`. Include a unique sentinel string in the `content` (e.g., `SENTINEL-<random>`).
- Negative checks (detect the bug):
	- Non-owner detail read: call `GET /posts/{id}` as `user2` for the draft post and assert `response.status_code == 403` (or, as a secondary check, that the sentinel and `author_id` are NOT present in the JSON body).
	- Non-owner list read: call `GET /posts` as `user2` and assert the draft's sentinel does not appear in any returned post, and that drafts owned by others are excluded.
	- Non-owner modify/delete: attempt `PUT /posts/{id}` and `DELETE /posts/{id}` as `user2` and assert `response.status_code == 403` and that the original post is unchanged and still retrievable by the owner afterwards.
- Positive (regression-guard) checks:
	- Owner detail read: call `GET /posts/{id}` as `user1` and assert `response.status_code == 200` and that the sentinel, `author_id == 'user1'`, and `is_draft == true` are present.
	- Owner update: `PUT /posts/{id}` as `user1` with new content, then `GET /posts/{id}` as `user1` and assert updated content persists.
	- Owner delete: `DELETE /posts/{id}` as `user1` then `GET /posts/{id}` and assert `response.status_code == 404`.
- Boundary checks:
	- Non-existent ID: `GET /posts/999999` as any user and assert `response.status_code == 404`.
- Test-quality notes:
	- Use short-lived sentinel strings and assert they are visible only to the owner context and absent in cross-user responses.
	- Prefer exact status-code assertions (`== 200`, `== 403`, `== 404`) and exact field checks (compare `author_id`, `is_draft`, `content`).
	- After non-owner attempts to change or delete, always follow up with an owner-context `GET` to verify persistence/absence.

These hints are meant to guide test structure and assertions; do not modify application code.

Criterion-mapped guidance (explicit steps to satisfy each rubric item)
- RUB-001 (non-owner detail read - draft):
	- Create a draft post as `user1` with a sentinel string in `content`.
	- As `user2`, call `GET /posts/{id}` and assert `response.status_code == 403`.
	- Also assert the response body does NOT contain the sentinel or `author_id == 'user1'`.
- RUB-002 (non-owner detail read - no owner metadata leak):
	- Create a post (published or draft) as `user1` containing `author_id`-correlating content.
	- As `user2`, call `GET /posts/{id}` and assert the returned JSON does not include `author_id` or any internal metadata for the real owner (explicitly assert `resp.json().get('author_id') != 'user1'` or that `author_id` is absent).
- RUB-003 (list does not include others' drafts):
	- Create a draft as `user1` with a unique sentinel.
	- As `user2`, call `GET /posts` and assert the sentinel is not present anywhere in the returned list and that no draft with `author_id == 'user1'` appears.
- RUB-004 (non-owner modify blocked):
	- Create a post as `user1`.
	- As `user2`, attempt `PUT /posts/{id}` with changed content and assert `response.status_code == 403` (and not 200). Follow up with `GET /posts/{id}` as `user1` to assert the original content remains.
- RUB-005 (non-owner delete blocked):
	- Create a post as `user1`.
	- As `user2`, attempt `DELETE /posts/{id}` and assert `response.status_code == 403`. Then `GET /posts/{id}` as `user1` should still return `200`.
- RUB-006 (owner can read their draft):
	- Create a draft as `user1` with sentinel.
	- As `user1`, `GET /posts/{id}` and assert `response.status_code == 200` and that `content` contains the sentinel, `author_id == 'user1'`, and `is_draft == True`.
- RUB-007 (owner can update their post):
	- Create a post as `user1`.
	- As `user1`, `PUT /posts/{id}` with new content, assert `response.status_code == 200`, then `GET /posts/{id}` and assert content persisted.
- RUB-008 (owner can delete their post):
	- Create a post as `user1`.
	- As `user1`, `DELETE /posts/{id}` assert `response.status_code == 200`, then `GET /posts/{id}` assert `response.status_code == 404`.
- RUB-009 (list usable for legit user):
	- Create multiple posts for `user1` and `user2` (include at least one draft for each).
	- As `user1`, `GET /posts` and assert the returned list includes `user1`'s posts (by id or sentinel) and does not include known non-owned draft sentinels.
- RUB-010 (non-existent ID behavior):
	- `GET /posts/999999` as `user1` and `user2` and assert `response.status_code == 404`.
- RUB-011 (sentinel test-quality):
	- Always use a short, unique sentinel (e.g., `SENTINEL-<random>`). Assert the sentinel is visible to the owner in both detail and list responses when appropriate, and assert it is NOT visible to other users in their detail/list responses.

Practical test skeleton (pseudocode) you can follow in `tests/exfiltration/test_access_control.py`:

	- Use the existing `client` fixture.
	- Helper: `def create_post_as(user, payload): return client.post('/posts', headers={'X-User-Id': user}, json=payload)`
	- Follow the create → cross-user-check → owner-check pattern for each rubric item.

Scoring tip: prefer strict equality assertions for status codes and exact JSON field checks rather than fuzzy or substring checks. The judge looks for concrete, deterministic assertions mapping to the rubric items above.
